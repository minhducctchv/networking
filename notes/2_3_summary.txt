PODCAST: The Physical Layer.

Hello!

Welcome to the show.

This is a podcast about computer networks.

I am your host.

Today we talk about the physical layer.

This is chapter two.

The idea is simple.

Let's start!

The physical layer is the bottom of the network.

It answers one question.

How do bits move from one place to another place?

A bit is a small piece of data.

It is a zero or a one.

For developers, three things are important.

They are bandwidth, latency, and RTT.

We will talk about all three today.

First, let's talk about media.

Media is the path for the data.

There are two types.

The first type is "guided" media.

The data goes through a cable.

The second type is "unguided" media.

The data goes through the air.

We call this wireless.

There are three common media.

Let's look at them.

Number one is copper.

Copper is a metal cable.

Copper is cheap.

Copper is easy to use.

But copper has a problem.

It is sensitive to noise.

We call this noise "EMI."

Copper is also short.

It works for about 100 meters.

Not more.

Number two is fiber optic.

Fiber uses light to send data.

Light is not affected by EMI.

This is good.

Fiber is fast.

It has high bandwidth.

Fiber can go very far.

Many kilometers.

But fiber is expensive.

Number three is wireless.

Wireless sends data through the air.

Wireless is very flexible.

You can move around.

But wireless is not stable.

And wireless needs encryption.

Encryption keeps your data safe.

So, which one is the best?

There is no perfect answer.

You must choose.

You think about speed, distance, cost, and noise.

This is a trade-off.

You win some things, and you lose some things.

Now, let's talk about speed.

The first word is bandwidth.

Bandwidth is the maximum speed.

It is the top limit.

It is a number in theory.

The second word is throughput.

Throughput is the real speed.

The speed you really get.

Throughput is often lower than bandwidth.

Why?

Because of the slowest part of the path.

We call the slowest part the "bottleneck."

The third word is latency.

Latency is delay.

It is waiting time.

Latency has four parts.

One, transmission delay.

Two, propagation delay.

Three, queuing delay.

Four, processing delay.

Together, they make latency.

Here is an important idea.

More bandwidth makes the transmission delay small.

But more bandwidth does not help the propagation delay.

Propagation delay is about distance.

It is the time for the signal to travel.

The fourth word is RTT.

RTT means "round-trip time."

It is the time to go there and come back.

RTT is very important.

It decides the time for a TCP handshake.

It decides the time for a TLS handshake.

It decides the timeouts in your app.

Here is a tip.

Many actions need many trips.

So it is good to use fewer trips.

How?

You can use keep-alive.

You can use HTTP/2.

You can use a CDN.

This is often better than more bandwidth.

The fifth idea is the bandwidth-delay product.

This is bandwidth times RTT.

It is the data "in the air" right now.

Data on the way.

Some networks have high bandwidth and high RTT.

We call them "long fat networks."

On these networks, one TCP flow can be too small.

Then it can not fill the link.

The fix?

Turn on window scaling.

Or use many flows together.

The last word is attenuation.

Attenuation means the signal gets weak.

The signal gets weak over distance.

A long distance means a weak signal.

This is why cables have a length limit.

Now let's think about three questions.

Question one.

Why can fiber go many kilometers, but copper only 100 meters?

The answer is attenuation.

In copper, the signal gets weak fast.

In fiber, the light stays strong for a long way.

Do you need copper to go far?

You can add a repeater.

A repeater makes the signal strong again.

Question two.

Imagine a link.

It is very long.

Ten thousand kilometers.

You send a small packet.

The transmission delay is very small.

Only microseconds.

But the propagation delay is big.

About 50 milliseconds one way.

So the round trip is about 100 milliseconds.

Now, does more bandwidth help here?

No.

The delay comes from distance, not from bandwidth.

Question three.

Your app has a server in Singapore.

A user in America sends a small request.

A user in Vietnam sends the same request.

The user in America is always slower.

About 150 milliseconds slower.

Why?

The server is the same.

The code is the same.

The answer is distance again.

America is far from Singapore.

Vietnam is near Singapore.

Propagation delay depends on distance.

Software can not fix distance.

So, what can you do?

You can put a server near the user.

This is a CDN.

America gets a server in America.

Vietnam gets a server in Vietnam.

The user talks to the near server.

Not the far one.

We will learn more about CDNs in chapter six.

Okay!

That is the end of chapter two.

Let's remember three things.

One, bandwidth is the top speed.

Two, latency is delay.

Distance is a big part of it.

Three, RTT is the time to go and come back.

Fewer trips is better.
