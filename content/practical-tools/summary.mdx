# Tóm tắt Chương 9: Practical Tools & Troubleshooting

Chương này biến kiến thức của cả khóa thành kỹ năng quan sát và tạo ra kết nối mạng thật. Dưới đây là các khái niệm chính theo từng bài, sau đó là bài tập ôn tập kết hợp.

## Các khái niệm chính

### Phân tích gói tin

- **Capture filter** (BPF) chạy lúc bắt, gói không khớp **mất vĩnh viễn**; **display filter** chạy lúc xem, **không xóa** dữ liệu.
- **tcpdump và Wireshark dùng chung BPF**; **display filter là cú pháp riêng của Wireshark**. Nguyên tắc: **bắt rộng, hiển thị hẹp**.

### Công cụ chẩn đoán

- **ping** đo tới đích + RTT + packet loss; **traceroute** lộ từng hop bằng TTL.
- **nslookup**/**dig** phân giải DNS; **netstat**/**ss** soi socket local (`-tuln` cho `LISTEN`, `-tnp` cho tiến trình sở hữu).

### Networking từ code

- Mọi call mạng có thể fail; **timeout mặc định thường vô hạn** trong `fetch`/`requests`/Go `DefaultClient` — luôn set tường minh.
- **Timeout tổng vs. timeout từng giai đoạn** (dial, TLS, response header) — Go `http.Transport` cho control chi tiết.
- **Tái dùng client instance** để tận dụng connection pool và Keep-Alive.
- **UDP** đơn giản, không handshake, không đảm bảo; **SSE** cần read timeout vô hạn.

### curl Cookbook

- **`-w` format với `%{time_*}`** đo latency từng giai đoạn (dns/connect/appconnect/starttransfer/total).
- **`--resolve host:port:ip`** ép IP cho hostname; **`--http1.1|2|3`** ép phiên bản.
- Test TLS/mTLS: **`--cacert`**, **`--cert/--key`**, **`-k`** (chỉ debug).
- **`--connect-timeout` + `--max-time` + `--retry`** cho request robust.

### Phương pháp troubleshooting

- Gỡ lỗi = **thu hẹp có hệ thống**: kiểm tra **bottom-up** theo tầng, **divide-and-conquer**; phép thử "tốt" cũng loại nhánh nguyên nhân.
- Dấu vân tay: DNS (ping IP OK, ping tên fail), đường truyền (mất gói + traceroute), dịch vụ (`Connection refused` vs `timed out`).

## Bài tập ôn tập

1. **(Tình huống — kết hợp "Networking từ code" + "curl Cookbook" + "Phương pháp troubleshooting")** Node.js service của bạn báo P99 latency = 2500ms khi gọi API bên ngoài, dù chính API đó chạy `curl -w "time_total"` cho `250ms`. Chẩn đoán khả năng: (a) DNS chậm ở node; (b) không tận dụng Keep-Alive; (c) không có timeout nên có request treo. Viết một lệnh curl với `-w` để xác nhận DNS và TLS đều nhanh, và đề xuất code fix cụ thể trong Node.js.

2. **(Thực hành — kết hợp "Phân tích gói tin" + "curl Cookbook")** Bạn muốn kiểm chứng bằng chứng vật lý rằng curl thực sự dùng HTTP/3 qua QUIC/UDP. Viết một lệnh `tcpdump` bắt lưu lượng tới một web server và một lệnh `curl --http3` để tạo request. Trong output tcpdump, đặc điểm gì cho thấy đây là UDP thay vì TCP?

3. **(Tình huống — kết hợp "Công cụ chẩn đoán" + "Networking từ code")** Ứng dụng Python `requests.get("https://api.example.com/data")` treo và không trả lời sau 2 phút. Bạn nghi ngờ server sống nhưng chậm. Trên máy chạy app: (a) dùng `dig` để kiểm DNS resolve; (b) dùng `curl` với timeout để kiểm server có nhận connection không; (c) dựa vào kết quả (a)/(b), sửa code Python thế nào để không treo mãi?

4. **(Lý thuyết — kết hợp "Phân tích gói tin" + "Phương pháp troubleshooting" + "Networking từ code")** Nguyên tắc "**bắt rộng, hiển thị hẹp**" trong packet analysis song song với nguyên tắc **"timeout ngắn ở connect, dài ở read"** trong client code ra sao? Cả hai đều thể hiện tinh thần "giữ tùy chọn mở, quyết định muộn khi có dữ liệu". Hãy giải thích qua một tình huống debug cụ thể.

1. Nhiều khả năng: (b) Node service tạo `fetch()` mới mỗi call → không tận dụng Keep-Alive → mỗi request tốn full handshake TLS. Xác nhận nghi ngờ bằng curl:

   ```bash
   curl -w "dns: %{time_namelookup}\nconnect: %{time_connect}\ntls: %{time_appconnect}\ntotal: %{time_total}\n" \
        -o /dev/null -s https://api.example.com/data
   ```

   Nếu `time_namelookup` < 50ms và `time_appconnect - time_connect` < 200ms, DNS và TLS không phải thủ phạm. Fix trong Node.js: dùng `undici.Agent` với `keepAliveTimeout` hoặc `Pool` để tái dùng kết nối:

   ```javascript
   const { Agent } = require("undici");
   const dispatcher = new Agent({ keepAliveTimeout: 60_000, connections: 20 });
   // Tái dùng trong mọi call:
   await fetch(url, { dispatcher, signal: AbortSignal.timeout(3000) });
   ```

   Đồng thời thêm `AbortSignal.timeout(...)` để không treo mãi.

2. Lệnh:

   ```bash
   $ sudo tcpdump -i any -n 'host cloudflare.com and udp port 443' &
   $ curl --http3 -sI https://cloudflare.com/
   ```

   Trong output tcpdump, mỗi dòng bắt đầu bằng `IP <src>.<port> > <dst>.443: UDP, length <n>` — chữ **UDP** trong dòng thay vì `Flags [S]`/`[.]` như TCP. Không có three-way handshake, không có cờ SYN/ACK, chỉ các datagram UDP đi lại (bên trong đó là QUIC payload đã mã hóa).

3. (a) `dig api.example.com A` — nếu nhận IP hợp lệ trong < 100ms, DNS OK. (b) `curl --connect-timeout 3 --max-time 10 -sI https://api.example.com/data` — nếu nối được nhưng response chậm/hang, biết được server nhận request. Nếu `Connection refused` thì server không lắng nghe/firewall chặn; nếu `timed out` thì gói không tới. (c) Sửa Python: thêm `timeout=(3, 10)` tường minh:

   ```python
   try:
       r = requests.get("https://api.example.com/data", timeout=(3, 10))
   except requests.exceptions.Timeout:
       # Xử lý timeout: log, retry với backoff, hoặc trả về default
       pass
   ```

Cả hai đều là "**capture nhiều, quyết định sau**": (i) Packet capture rộng — nếu display filter sai giả thuyết, dữ liệu vẫn còn để thử filter khác; nếu capture filter hẹp sai, dữ liệu mất luôn phải bắt lại. (ii) Timeout: connect nên **ngắn** vì thất bại nhanh giúp phát hiện sớm mạng có vấn đề (bắt "đường không tới" ngay); read nên **dài hơn** vì server chậm không phải là lỗi mà là dữ liệu chưa tới đủ — cắt sớm thì mất kết quả có thể sắp tới. Tình huống cụ thể: debug API chậm. Bắt gói với `tcp port 443` (rộng), rồi trong Wireshark filter `tcp.flags.syn == 1` để xem handshake — nếu chưa đủ, đổi filter `http.request` để xem request thật, không phải capture lại. Song song, client cấu hình `--connect-timeout 2 --max-time 30` — connect nhanh phát hiện lỗi mạng, read dài chờ server có thể chậm nhưng vẫn hợp lệ. Cả hai tránh trả lời quá sớm khi thiếu thông tin.
