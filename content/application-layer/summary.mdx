# Tóm tắt Chương 6: Application Layer

Chương này đi qua các protocol mà lập trình viên dùng trực tiếp: DNS, HTTP các phiên bản, giao thức realtime, HTTP caching và CDN, cùng SMTP/FTP/SSH. Dưới đây là các khái niệm chính theo từng bài, sau đó là bài tập ôn tập kết hợp nhiều bài.

## Các khái niệm chính

### DNS

- **DNS** biến domain name thành địa chỉ IP, chạy trên **port 53** (UDP mặc định, TCP khi phản hồi lớn).
- Không gian tên **root → TLD → authoritative**; chỉ chặng stub đến recursive resolver là **recursive query**, còn resolver đi hỏi root/TLD/authoritative là **iterative query**.
- **TTL** quyết định thời gian cache; bốn record cốt lõi: **A**, **AAAA**, **CNAME**, **MX**.

### HTTP & HTTPS

- HTTP là **request/response, stateless**, chạy trên TCP **port 80**; HTTPS trên **port 443**.
- Request: **method + path + version**, header, body. Response: **status line** với 5 nhóm status code, header, body.
- **HTTPS = HTTP + TLS**: xác thực qua certificate do CA ký, thỏa thuận session key đối xứng.

### HTTP/2 & HTTP/3

- **HTTP/2** giữ nguyên ngữ nghĩa HTTP nhưng đổi vận chuyển: **binary framing + streams multiplex** trên một TCP + **HPACK** nén header; negotiate qua **ALPN**.
- HTTP/2 vẫn dính **TCP head-of-line blocking**; **HTTP/3 = HTTP + QUIC** (trên UDP) khắc phục bằng stream độc lập, handshake 1-RTT/0-RTT, và **connection migration**.
- Thực tế: bật cả ba song song qua CDN; browser chọn HTTP/3 khi được, fallback HTTP/2.

### WebSocket, SSE & gRPC

- **WebSocket** (`wss://`): kênh **full-duplex** trên TCP, bắt đầu bằng HTTP upgrade; không có auto-reconnect. Hợp cho chat, gaming.
- **SSE** (`text/event-stream`): server đẩy event một chiều qua HTTP; **auto-reconnect có sẵn**. Hợp cho feed, notification.
- **gRPC** trên **HTTP/2 + protobuf**: 4 kiểu gọi (unary, server/client/bidi streaming); browser cần **gRPC-Web** qua proxy.

### HTTP Caching & CDN

- **`Cache-Control`** điều khiển cache: **`no-cache` = validate, `no-store` = không lưu**; `private`/`public`, `max-age`, `s-maxage`, `immutable`.
- **`ETag` + `If-None-Match`** cho validation không tải lại body; server trả **`304 Not Modified`**.
- **CDN** đưa cache tới gần user qua **anycast + edge nodes**; pattern chuẩn cho asset tĩnh: **hash trong tên file + `immutable, max-age=1y`**.

### SMTP, FTP & SSH

- **SMTP** gửi/chuyển email: **port 25** relay, **port 587** submission có xác thực.
- **FTP**: **port 21** control + **port 20** data; nên thay bằng SFTP/FTPS.
- **SSH** (**port 22**): kênh mã hóa cho shell từ xa, SCP/SFTP, port forwarding.

## Bài tập ôn tập

1. **(Lý thuyết — kết hợp "DNS" + "HTTP & HTTPS" + "HTTP/2 & HTTP/3")** Khi bạn gõ `https://shop.example.com`, hãy mô tả theo thứ tự: (a) DNS resolve, (b) chọn phiên bản HTTP qua ALPN, (c) handshake TLS. Với HTTP/3, bước nào được rút ngắn so với HTTP/2, và tại sao?

2. **(Tình huống — kết hợp "WebSocket, SSE & gRPC" + "HTTP/2 & HTTP/3")** Bạn xây tính năng thông báo push cho web app (server đẩy thông báo mới về từng user). Bạn cân nhắc SSE và WebSocket. Nếu backend đã chạy trên HTTP/2, cách nào phù hợp hơn và vì sao? Nếu ngày mai chuyển sang HTTP/3, có ảnh hưởng gì tới lựa chọn không?

3. **(Tình huống thực tế — kết hợp "HTTP Caching & CDN" + "DNS")** Bạn deploy phiên bản mới của SPA nhưng nhiều user vẫn thấy giao diện cũ dù đã hard-refresh. Kiểm tra: bundle JavaScript có tên `app.js` (không hash), header response là `Cache-Control: max-age=86400`. Chẩn đoán vấn đề và đề xuất pattern để không lặp lại. Liên hệ tới TTL của DNS: có tương đồng nào giữa "TTL DNS cao gây khó migrate IP" và "max-age cao gây khó deploy version mới"?

4. **(Tình huống — kết hợp "HTTP/2 & HTTP/3" + "HTTP Caching & CDN")** Trang thương mại điện tử của bạn phục vụ hàng triệu request/ngày. Bạn cần giảm chi phí origin và cải thiện latency. Hãy đề xuất kết hợp: (a) HTTP version nào bật ở CDN; (b) chiến lược cache cho HTML gốc, asset tĩnh (JS/CSS/ảnh), và API response cá nhân hóa (giỏ hàng người dùng). Giải thích lựa chọn cho từng nhóm.

1. (a) **DNS resolve**: stub gửi recursive query tới resolver; resolver đi hỏi root → TLD `.com` → authoritative của `example.com` để lấy A/AAAA của `shop.example.com`. (b) **ALPN**: trong lúc TLS handshake, client gửi danh sách protocol hỗ trợ (`h3`, `h2`, `http/1.1`); server chọn cái phù hợp — thường là `h3` hoặc `h2` nếu client hỗ trợ. (c) **TLS handshake**: client xác thực certificate của server và thỏa thuận session key. Với **HTTP/3 (QUIC)**, bước (c) được **gộp** với bước thiết lập kết nối UDP thay vì tách rời như TCP + TLS 1.3 — chỉ tốn 1 RTT (hoặc 0-RTT nếu resume phiên); HTTP/2 tốn 1 RTT cho TCP + 1 RTT cho TLS = 2 RTT trước khi HTTP request đi được.

2. Với thông báo push một chiều (server → user), **SSE** phù hợp hơn: đơn giản, auto-reconnect có sẵn, đi qua HTTP proxy/CDN như request thường. WebSocket là overkill vì không cần chiều ngược. HTTP/2 vs HTTP/3 không đổi lựa chọn: SSE là HTTP thường nên chạy trên bất kỳ phiên bản HTTP nào — chuyển từ H2 sang H3 chỉ **cải thiện hiệu năng** (connection migration cho mobile) mà không phải viết lại code SSE.

Vấn đề: bundle không có hash trong tên (`app.js`), và `Cache-Control: max-age=86400` khiến browser và CDN cache 24 giờ; deploy đổi nội dung nhưng URL vẫn giống, cache không biết đã có bản mới, nên serve bản cũ tới 24 giờ. Pattern: **thêm hash trong tên file** (`app.a3f2c891.js`) + `immutable, max-age=1y`; HTML gốc thì `no-cache` hoặc `max-age=0` để browser luôn kiểm tra. **Tương đồng với DNS TTL**: cả hai đều là "cam kết dữ liệu không đổi trong X thời gian" — TTL DNS cao giúp giảm truy vấn nhưng làm chậm việc thay IP; `max-age` cao giúp giảm request nhưng làm chậm việc thay nội dung. Kỹ thuật xử lý cũng giống: đổi định danh (URL với hash, hostname mới) thay vì chờ TTL/max-age hết.

4. (a) **Bật cả HTTP/2 và HTTP/3** ở CDN; browser tự chọn (thường ưu tiên H3, fallback H2 nếu UDP bị chặn). (b) Chiến lược cache: **HTML gốc** — `Cache-Control: no-cache` hoặc `max-age=0, must-revalidate`; browser luôn hỏi lại (dùng ETag để tiết kiệm băng thông khi chưa đổi) — cần cập nhật kịp thời để trỏ tới bundle mới. **Asset tĩnh** — pattern hash trong tên + `public, max-age=31536000, immutable`; CDN cache tối đa, browser cũng cache mãi, deploy đổi hash → URL khác. **API response cá nhân hóa (giỏ hàng)** — `Cache-Control: private, no-store` (hoặc `private, max-age=0`); tuyệt đối không cache ở CDN, chỉ browser của chính user, và không bao giờ mix giữa user vì gắn với `Authorization`/session cookie.
